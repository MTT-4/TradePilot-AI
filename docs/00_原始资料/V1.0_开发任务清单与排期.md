# TradePilot AI · V1.0 开发任务清单与排期（AI 代理驱动）

> **背景：** 创始人无开发团队。**Codex 主力自主开发，Claude（Claude Code）审核 + 续作 + 修复。** 本文是给两个代理照着执行/审核的施工图。
> **配套：** 《V1.0 MVP 产品设计书》为需求依据，本文为实现依据。
> **版本冻结：** 本版按最新决策执行：**轻量短视频成片 / TTS / 自动字幕 → V1.5；复杂多租户代运营后台 → M7；除这两类外，V1.0 主闭环功能必须全做。**

---

## 0. 工作方式（Codex 建 → Claude 审 → 你验收）

1. 建一个 Git 仓库，把《设计书》+ 本文放进 `/docs` 作为两个代理的共同上下文。
2. **一次只派一个任务（或一个里程碑），不要让代理一口气全做完不验。** 每个任务把“描述 + 验收标准”原样喂给 Codex。
3. **Codex** 实现并提交 PR/分支。
4. **Claude** 审核该 PR：跑起来、读代码、重点查安全/租户隔离/隐私路由（见第 4 节），通过则合并，不通过则直接修或退回 Codex。
5. **你**：拍板决策、按每个里程碑的“验收”确认能用、保管所有密钥（OpenAI/Google）。代理不持有生产密钥。

> 真相提醒：纯自主代理建一个“多租户 + 隐私敏感”的 SaaS，最容易在权限、租户隔离、密钥处理上犯**隐蔽且代价高**的错。所以 Claude 审核不是可选项；**接入任何真实客户数据前，至少做一次人工安全审查。**

## 1. 技术栈（已选定，避免代理乱选）

| 层 | 选型 | 理由 |
|---|---|---|
| 全栈框架 | **Next.js (App Router) + TypeScript** | 前后端一体、代理训练数据多、好维护 |
| UI | **Tailwind + shadcn/ui** | 与高保真原型一致 |
| 数据库 | **PostgreSQL + pgvector** | 关系数据 + 向量库一库搞定 |
| ORM | **Prisma** | schema 即文档，迁移可控 |
| 异步任务 | **BullMQ + Redis** | 向量化/渲染/发布等耗时任务 |
| 鉴权 | **Auth.js** | 邮箱密码 + 2FA + 会话 |
| 对象存储 | **S3 兼容**（本地用 MinIO） | 上传与素材 |
| 模型网关（自研模块） | OpenAI API · Google Cloud Translation（含 glossary）· **本地 llama.cpp `llama-server`（OpenAI 兼容端点）跑 Qwen + bge-m3** | 按数据敏感度路由 |
| 部署 | 开发在你的 Mac → 上线迁自有云 GPU/服务器 | 见 M7 |

## 2. V1.0 范围冻结

### 2.1 V1.0 必须完成

V1.0 不是只做 Demo，必须完成完整外贸获客闭环：

知识库导入 → AI 建站 → SEO/GEO → 9 平台内容包 → 人工发布清单 → 追踪链接 → 表单/邮件询盘 → CRM → 本地 Qwen 首响草稿 → 规则打分 → 看板归因 → 工作台下钻。

### 2.2 V1.0 明确不做，后移版本

1. **轻量短视频成片 / TTS 配音 / 自动字幕 → V1.5**  
   V1.0 仍要覆盖视频平台，但只输出：视频脚本、分镜建议、封面图、标题、文案、标签、规格校验、追踪链接、发布清单。**不执行成片渲染、不生成 TTS、不自动加字幕。**

2. **复杂多租户代运营后台 → M7**  
   V1.0 前置阶段仍必须做基础多租户隔离、tenant_id、对象存储/向量隔离、审计日志。复杂的“一个服务商账号管理多个客户租户、跨租户切换、套餐/用量后台、平台运营后台”统一放到 M7，不阻塞 M0–M6 闭环验证。

## 3. 里程碑与排期（概览）

> 排期给区间。**瓶颈不是写代码，是你审核与决策的节奏**——代理写得快，你验得慢就慢。下列为“含审核往返”的大致 elapsed 时间。

| 里程碑 | 内容 | 预估 | 依赖 |
|---|---|---|---|
| **M0 地基** | 脚手架 · 基础租户隔离 · 鉴权 · 模型网关 · 追踪服务 · 应用外壳 · 四份契约 + E2E 骨架 | 8–12 天 | — |
| **M1 知识库** | 上传 → bge-m3 入库 → 审核 → 检索 | 4–6 天 | M0 |
| **M2 建站** | 对话生成 → 多语言 → SEO/GEO → 发布 → 自动补全草稿 | 6–9 天 | M0,M1 |
| **M3 设计/内容包** | 选题对话 → 图文生成 → 视频脚本/分镜 → 9 平台适配 → GPT 校正 → 追踪链接 → 清单 | 6–9 天 | M0,M1 |
| **M4 询盘→CRM** | 表单/邮件归集 → 去重归因 → 最小 CRM → 管道 | 4–6 天 | M0 |
| **M5 首响/看板** | 本地 Qwen 首响 → 规则打分 → 看板归因 | 3–5 天 | M1,M4 |
| **M6 控制台整合** | 闭环图 · 可下钻 · 通知中心 · HITL | 3–4 天 | M2–M5 |
| **M7 硬化与复杂多租户上线** | 完整 RBAC · 复杂代运营后台 · 平台运营后台 · 安全审计 · 本地模型迁云 · 端到端验收 | 6–9 天 | 全部 |

**关键路径：** M0 必须先做扎实（尤其基础租户隔离 + 模型网关 + 追踪服务），它是其它一切的地基。总计约 **6–10 周** elapsed，M7 可按上线压力单独拉长。

## 4. 详细任务清单

> 标 **[关键审核]** 的任务，Claude 必须重点审、不得放行带病代码。

### M0 · 地基
- **T0.1 项目脚手架** — Next.js+TS+Tailwind+shadcn，目录结构、lint/format、`.env` 管理、Git。
- **T0.2 数据模型与 Prisma** — 基础 schema，**所有业务表强制带 `tenant_id`**，迁移脚本。**[关键审核]**
- **T0.3 基础多租户中间件** — 请求级租户上下文，所有查询强制按 `tenant_id` 过滤，杜绝越权读取。复杂代运营跨租户切换不在 M0 做，放 M7。**[关键审核]**
- **T0.4 鉴权与角色骨架** — Auth.js 邮箱密码 + 2FA + 会话；RBAC 角色枚举（Owner/Admin/运营/业务员/只读）。**[关键审核]**
- **T0.5 模型网关** — 统一 `LLM.invoke / Translate / Embed` 接口；**按数据敏感度路由**（公开营销→OpenAI、公开翻译→Google、含个资/仅内部→本地 Qwen）；接 OpenAI、Google 翻译、本地 llama.cpp 端点（地址走配置）；写审计日志。**[关键审核：隐私路由不得误走第三方]**
- **T0.6 DataClassifier 隐私分类器** — 第三方模型调用前必须分类；检测到姓名、邮箱、电话、客户询盘正文、内部案例、报价、合同、未公开资料、标记“仅内部”的知识块，一律强制本地 Qwen；本地 Qwen 不可用时不得 fallback 到 OpenAI/Google，只能返回“隐私模型暂不可用”。**[关键审核]**
- **T0.7 追踪链接服务** — 生成短链/UTM、记录点击、解析来源归因 API。需防 open redirect、短链不可枚举、服务端 campaign_id 为准、bot 点击可过滤。闭环主线，**最先联调**。
- **T0.8 异步队列 + 对象存储** — BullMQ+Redis、MinIO/S3 接入，按租户 bucket prefix 隔离。
- **T0.9 应用外壳** — 侧栏导航 + 顶栏 + 通知中心骨架，套用高保真原型样式，正式实现拆成 Next.js 组件，不直接沿用静态 HTML。
- **T0.10 数据库 Schema 契约** — 在 `/docs/schema-contract.md` 明确核心表、字段、索引、状态枚举、`tenant_id` 隔离策略、对象存储路径、向量 namespace。**[关键审核]**
- **T0.11 API 契约** — 在 `/docs/api-contract.md` 明确工作台、知识库、建站、内容包、CRM、HITL、通知中心所需 API、字段、错误码、权限。**[关键审核]**
- **T0.12 端到端测试骨架** — 建立 Playwright + API test，覆盖登录、租户隔离、隐私路由、追踪链路、HITL 闸门。**[关键审核]**
- **T0.13 种子数据** — 提供两个租户、样本公司、样本知识库、样本内容包、样本询盘，用于前后端联调与验收。

- **验收：** 能注册/登录（带 2FA）；两个租户数据互不可见；模型网关按敏感度正确路由并留日志；含隐私请求本地 Qwen 不可用时不得走第三方；生成一条追踪链接、点击能被记录并归因；Schema/API/UI 数据契约与 E2E 测试骨架已提交；种子数据可一键初始化。

### M1 · 知识库
- **T1.1 上传与解析** — PDF/Word/Excel/URL 抓取 → 文本提取。
- **T1.2 切块 + 元数据** — 切块并打标（产品/市场/语言/来源/**敏感级：仅内部 vs 可公开**）。
- **T1.3 向量化入库** — bge-m3（llama.cpp embedding 端点）→ 写入 pgvector，**按租户隔离 namespace**；异步处理 + 进度。**[关键审核：隔离]**
- **T1.4 检索接口** — 语义 + 关键词混合，按 `tenant_id` + 产品/市场/语言/敏感级过滤；**抗幻觉护栏（无依据不编造）**。
- **T1.5 知识审核页** — 知识卡片确认/修正/标“已核准”/溯源；支持手动补问答对。
- **验收：** 拖文件→看到进度→可检索；审核页能改并标已核准；“仅内部”资料只允许进入本地 Qwen 路径；公开营销生成不得引用内部知识块。

### M2 · 建站智能体
- **T2.1 对话式生成** — 半结构化引导（5–6 必填项），取知识库，模板填充，单次 ≤3 页。
- **T2.2 多语言与 URL** — OpenAI 出源语 + Google 翻译（glossary 锁型号品牌）；**阿语 RTL**；每语种独立 URL + hreflang。
- **T2.3 SEO/GEO 注入** — TDK、sitemap、hreflang、JSON-LD、顶部快答块、FAQ、robots 放行 AI bot、关键内容 SSR、**OG/VK 社媒分享卡**。
- **T2.4 站点管理台** — 列表/上下线/版本快照回滚/预览/**HITL 发布**；响应式渲染。
- **T2.5 AI 自动补全内容** — 新产品/认证/博客由知识库自动生成草稿，**每条需“确认上线”且可调整/编辑**。
- **验收：** 对话生成 EN/AR/RU 多语言站→预览→HITL 发布；切语言 URL 带后缀、阿语 RTL；SEO/GEO 基线生效；自动补全条目可确认上线/编辑。

### M3 · 设计 / 内容包
- **T3.1 人 + AI 设计对话窗口** — AI 给选题建议 + 用户逐平台调整文案、语气、规格、发布时间，实时改。
- **T3.2 图文生成** — 文生图、图生图、抠图换背景 + 品牌资料包；输出平台适配图、封面图、配套文案与标签。
- **T3.3 视频平台脚本与分镜（V1.0 必做）** — 为 Reels / TikTok / Shorts / VK Клипы / RuTube 输出视频脚本、分镜建议、封面图、标题、文案、标签、追踪链接和发布清单。
- **T3.4 V1.0 不做视频成片** — **不生成短视频成片、不做 TTS 配音、不做自动字幕。** 若任务涉及视频平台，只做脚本/分镜/封面/文案/规格校验；成片渲染、TTS、自动字幕进入 V1.5。
- **T3.5 平台规则库 + GPT 校正** — 9 平台（LinkedIn/Facebook/Instagram/Reels/TikTok/YouTube/Shorts/VK/RuTube）尺寸、比例、文案长度、标签、本地化、发布时间建议做成**可配置规则库**；GPT 按规则给建议并自动校正。视频相关规则只做脚本与发布规格校验，不执行剪辑或配音。
- **T3.6 内容包 + 清单** — 同一选题生成 9 平台内容包，**每份预埋追踪链接**；发布清单包含平台、内容类型、规格、文案、标签、发布时间、追踪链接、是否已发布；支持一键导出。
- **验收：** 选一个选题→生成 9 平台内容包→图文内容可用→视频平台有脚本/分镜/封面/文案/标签→GPT 校正生效→每条都有追踪链接→发布清单可导出。

### M4 · 询盘 → CRM
- **T4.0 入站邮件架构确认** — 系统邮箱格式、租户映射、入站 webhook、邮件解析、附件存储、去重规则、垃圾过滤、审计日志。
- **T4.1 表单归集** — 落地页表单 → 线索 + 用追踪链接解析来源（平台→内容）。
- **T4.2 邮件归集** — 专属系统邮箱接收 + 解析 + 去重（不连客户 IMAP）；邮件正文与附件默认按潜在隐私处理。
- **T4.3 最小 CRM** — 客户/联系人/商机阶段/跟进记录/来源字段；拖拽管道 + 到期提醒。
- **验收：** 表单与邮件两来源→统一线索池（去重 + 来源归因）→进 CRM 管道；线索能追溯到平台、内容、追踪链接。

### M5 · 首响 / 打分 / 看板
- **T5.1 AI 首响草稿** — **本地 Qwen** 生成多语言草稿（取知识库），**HITL 一键发送**，可编辑/模板。**[关键审核：走本地、不出境]**
- **T5.2 规则式打分** — A/B/C，规则可配置、可解释。
- **T5.3 看板** — 今日新询盘/待跟进/**来源分布（平台→内容→询盘数）**。
- **验收：** 新询盘→本地起草→人工一键发；看板正确显示来源归因；客户隐私内容无第三方模型调用日志。

### M6 · 控制台整合
- **T6.1 工作台闭环图 + 可下钻** — 节点/统计数字/智能体卡片点击进对应模块。
- **T6.2 通知中心 + HITL 队列** — 待处理询盘、首响草稿待审批、内容包待发布、网站改动待上线统一入口，点击直达。
- **T6.3 UI 数据契约** — 为工作台、知识库、建站、内容包、CRM、通知中心补 `/docs/ui-data-contract.md`，明确每个数字、按钮、列表对应 API 与字段。
- **T6.4 四角色实战验收** — 前端、后端、用户、运营按《V1.0 四角色实战跑通验收报告》第 6 节剧本完整跑一遍，验证“知识库→建站→内容包→追踪链接→询盘→CRM→首响→看板”不断链。

- **验收：** 原型里所有主要可点交互都接真实数据并正确跳转；HITL 动作有审批记录；四角色实战验收剧本完整通过。

### M7 · 硬化与复杂多租户上线
- **T7.1 RBAC 完整化** — 角色 × 操作授权，精细到模块和动作。**[关键审核]**
- **T7.2 复杂多租户代运营后台** — 一个服务商账号可管理多个客户租户；支持跨租户切换；所有跨租户操作必须留痕。**[关键审核]**
- **T7.3 平台运营后台** — 租户管理、套餐、Credits、模型配额、系统监控、告警。

- **T7.5 本地模型迁移** — Qwen/bge-m3 从你的 Mac 迁到自有云 GPU/服务器，仅改模型网关端点配置。
- **T7.6 端到端验收** — 按《设计书》第 8 节闭环线跑通；压一遍租户隔离、隐私路由、追踪链路。
- **接真实客户数据前：人工安全审查 / 渗透测试。**

## 5. Claude 重点审核清单（高危区，逐项必查）

1. **多租户隔离** — 每条查询都带 `tenant_id`；向量库/对象存储按租户分隔；构造两个租户做越权读取测试。
2. **隐私路由** — 含客户个资、询盘正文、报价、合同、内部案例或“仅内部”的请求**绝不**发往 OpenAI/Google；路由有 DataClassifier、兜底分支、审计日志可追。
3. **本地模型失败兜底** — 本地 Qwen 不可用时，隐私任务必须失败并提示，不得 fallback 到第三方模型。
4. **鉴权与密钥** — 2FA 生效；密码加盐哈希；第三方密钥不入代码库、不下发给前端、按租户限额。
5. **追踪链接完整性** — 建站表单 ↔ 内容追踪 ↔ CRM ↔ 看板四处来源 ID 贯通，不丢不串；防 open redirect；短链不可枚举；bot 点击可过滤。
6. **HITL 闸门** — 发布内容、网站上线、回复客户均需人工确认，无“静默自动对外”路径。
7. **V1.0 范围纪律** — 不允许 Codex 把短视频成片、TTS、自动字幕提前塞进 V1.0；不允许把复杂代运营后台提前阻塞 M0–M6。

## 6. 给两个代理的协作守则

- **Codex**：一次实现一个任务；严格按验收标准；不臆造需求；涉及密钥/对外发送/删数据/隐私路由的逻辑要显式标注，等审核。
- **Claude**：审核优先于产出；发现高危项直接拦截；可续写测试、补缺失的隔离/校验、修 Codex 的漏洞；每个里程碑出一份“审核结论 + 遗留问题”。
- **你**：每个里程碑只问一句——“按验收标准，它真能跑通吗？”跑通再进下一个。

## 7. 四角色实战跑通验收（前端 / 后端 / 用户 / 运营）

> 目的：避免“单模块看似完成，真实使用链路断掉”。M0 先补工程契约，M6 做最终四角色实战验收。

### 7.1 总判定

- **前端：条件通过** —— 页面与交互方向成立，但必须依赖 `/docs/ui-data-contract.md`、状态机、空态/失败态、HITL 统一组件。
- **后端：条件通过** —— 架构成立，但 M0 必须补 `/docs/schema-contract.md`、`/docs/api-contract.md`、E2E 测试骨架、种子数据。
- **用户：基本通过** —— 用户能完成导资料、建站、生成内容、收询盘、审批首响，但需要“下一步”向导式体验。
- **运营：基本通过** —— 内容发布、线索跟进、看板归因成立，但必须补负责人、计划时间、实际发布时间、发布状态、首响状态、追踪链接状态。

### 7.2 四角色端到端验收剧本

| 步骤 | 角色 | 动作 | 通过标准 |

| 6 | 前端 | 预览 EN/AR/RU | 阿语 RTL 正确，表单可用。 |
| 7 | 运营 | 确认上线 | 产生 HITL 审批记录。 |

### 7.3 开发闸门

- M0 不通过：不得进入 M1。
- 租户隔离测试不通过：不得接入真实客户数据。
- 隐私路由测试不通过：不得启用 OpenAI/Google。
- 追踪链路测试不通过：不得验收 M2/M3/M4/M5。
- HITL 测试不通过：不得允许任何对外动作上线。
- M6 四角色实战验收不通过：不得进入 M7 上线硬化。
