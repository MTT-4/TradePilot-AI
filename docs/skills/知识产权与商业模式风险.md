# TradePilot AI · 知识产权 / 专利 / 商业模式风险审查

> 免责：本文为信息梳理，非法律意见。具体落地前请由 IP 律师做 FTO（自由实施）检索与合同审查。
> 评级：🔴高　🟠中　🟢低/可控。

---

## 一、知识产权（IP）

🔴 **第三方大模型的商用与输出条款**：换不同模型时 ToS 差异很大。
- 通义千问 Qwen / bge-m3（本地）：确认所用权重版本的**商用许可**（部分版本对月活/营收有门槛或需授权）。
- DeepSeek / Moonshot / OpenAI（API）：确认①允许商用产品调用；②是否允许用其**输出训练/蒸馏自有模型**
  （多数禁止"用输出开发竞争模型"）；③输出的权利归属与可转授给你客户的范围。
- 风险点：把外部模型输出直接作为你 SaaS 卖给客户的"成品内容"，权利链要清楚。

🟠 **AI 生成内容的版权归属不确定**：AI 纯生成的营销文案/网站/回复，在多数法域**不享有版权或归属模糊**。
影响：你无法主张这些产物的独占版权，客户也难维权。建议合同里写明生成内容"按现状提供、版权状态不确定"。

🟠 **客户上传资料的权利边界**：客户传产品图/PDF/认证文件进 KB，你需获得"为其处理/生成派生内容"的授权。
建议服务协议加一条客户对上传内容的权利保证与授权许可（你只在租户内使用、不跨租户、不用于训练）。

🟠 **开源依赖许可合规**：现有及将引入的库（pdf-parse、mammoth、xlsx、cheerio、bullmq 等）多为 MIT/Apache，
风险低；但要避免引入 **AGPL/GPL** 传染性许可的库进核心服务。建议加一条 CI 许可证扫描。

🟠 **用客户数据微调/改进模型**：若未来拿客户询盘、知识库去微调，触发 IP + 隐私双重风险。
建议：默认不拿租户数据训练；要做必须每租户显式授权 + 脱敏 + 隔离。

🟢→🟠 **商标**："TradePilot" 名称可能与他人已注册商标撞车（该词较通用）。建议上线前做商标检索，
重点市场（中国/欧盟/美国）先查 in-use 与注册情况，避免改名成本。

## 二、专利

🟠 **侵权风险（FTO）**：外贸/CRM/AI 邮件领域专利布局密集（Salesforce、HubSpot 等持大量 CRM/营销自动化专利）。
你的"lead scoring、自动跟进、报价助手、邮件自动化"多为已知手法，**自创难，但要防踩他人已授权专利**。
建议：上线前做一次 FTO 检索，重点是 lead scoring 算法、邮件自动化流程、报价生成这几个高密度区。

🟠 **第三方数据工具的数据/专利权利**：ImportYeti（海关数据）、Similarweb（流量）、Apollo（B2B 资料）
的数据集本身受合同/数据库权利保护，**不是专利问题但是权利问题**（见第三节）。

🟢 **你自己可考虑的防御性专利**：若"本地隐私优先路由（敏感数据强制本地模型）+ HITL 审批闭环"在你的实现上
确有新颖性，可评估申请，作为差异化与防御资产。需专利代理评估新颖性/创造性，别自行假设可专利。

## 三、商业模式

🔴 **第三方 API 的 ToS 即生死线**：业务高度依赖外部服务，对方政策一变或封号即业务中断。
- WhatsApp Business：政策严格，**禁止未授权群发/营销骚扰**，违规封号。
- Gmail API：自动化发信受配额与反垃圾政策约束。
- Hunter/Apollo/ImportYeti：通常**禁止转售/再分发数据**——你把它们的数据做进 SaaS 卖给客户，
  很可能违反其 ToS。这是最容易被忽视的商业模式雷区。
- Stripe：合规与风控政策，违规冻结资金。
建议：每个外部依赖都做"ToS 合规映射 + 降级方案 + 多供应商备份"，别单点依赖。

🔴 **获客合规即法律风险**：自动开发信/群发触及 **GDPR（欧盟）、CAN-SPAM（美国）、PIPL（中国）**。
违规罚款高、损声誉。建议：线索工具默认人工确认后才可联系，内置退订/同意管理，禁止无差别群发。

🟠 **平台对 AI 错误输出的连带责任**：AI 报错价、错认证、错合规建议，客户据此对外承诺造成损失，
你作为平台方可能被追责。缓解：①HITL 留痕（已具备）；②所有价格/交期/认证标"需人工确认"；
③服务协议明确 AI 输出为辅助、最终责任在用户、限责条款。

🟠 **护城河偏浅**：若核心只是"调第三方 API + 通用 AI"，易被复制。
差异化应押在：本地隐私优先路由（合规卖点）、外贸行业 know-how、租户数据闭环、HITL 工作流，而非可替代的 API。

🟠 **模型成本与毛利**：依赖外部 LLM 时成本随用量波动、侵蚀毛利；本地模型摊薄但有硬件/运维投入。
建议：用量计费（`billing_usage_tool` + 已有 `ModelInvocation` 记录）把成本透传或分层定价。

🟠 **厂商锁定**：过度绑定单一模型/数据源会被议价或断供。model-gateway 已支持换模型（改 .env/策略），
这点架构上做得好，应保持"可插拔"作为商业韧性。

## 优先处理清单

1. 🔴 逐个外部依赖做 ToS 合规映射，重点确认 **数据转售/群发/输出训练** 三类禁止条款。
2. 🔴 获客模块内置同意/退订管理，默认人工确认，杜绝无差别群发。
3. 🔴 确认所用模型权重/API 的商用许可与输出权利链。
4. 🟠 上线前：商标检索 + 一次 FTO 专利检索（lead scoring / 邮件自动化 / 报价）。
5. 🟠 服务协议：客户上传授权、AI 输出免责限责、版权状态声明。
6. 🟠 CI 加开源许可证扫描，拦截 AGPL/GPL 进核心。
7. 🟢 评估"本地隐私路由 + HITL"的防御性专利价值（交专利代理判断）。
